# PRD: CI/CD Pipeline Health Monitor

> **Workshop Use:** Practice exercise for headless mode, GitHub Action integration, and building custom subagents. This PRD is specifically designed to exercise Act 4 of the workshop (Scaling the Machine).

## Problem

An engineering team runs 15 microservices with ~40 CI/CD pipelines across GitHub Actions. When builds fail, engineers get a Slack notification with a link to the log. They then spend 10-30 minutes reading through verbose build output to find the actual error. Flaky tests cause 20% of failures — engineers retry manually and waste time.

## Users

| Role | Need |
|:---|:---|
| **Developer** | Know immediately WHY a build failed — not just that it failed. Get a suggested fix. |
| **Tech Lead** | Identify chronically flaky tests and pipelines across all repos. Make data-driven decisions about what to fix. |
| **SRE** | Detect infrastructure-related failures (runner out of disk, npm registry timeout) vs. code-related failures. |

## Features

### 1. Failure Diagnosis Agent
- Triggered automatically when a GitHub Actions workflow fails
- Reads the full build log (often 5,000+ lines)
- Extracts: the actual error message, the failing step, the relevant file/line
- Classifies the failure:
  - `code_error` — compilation failure, type error, linting violation
  - `test_failure` — a specific test assertion failed
  - `flaky_test` — test passed on retry (detected by comparing with previous runs)
  - `infra_failure` — runner timeout, network error, registry unavailable
  - `config_error` — bad environment variable, missing secret, wrong Node version
- Posts a structured summary as a PR comment or issue comment

### 2. Auto-Fix Suggestions
- For `code_error`: read the failing file, understand the error, suggest a fix
- For `test_failure`: read the test and the code under test, explain what broke
- For `flaky_test`: identify the test, flag it, suggest `@retry` annotation or quarantine
- For `config_error`: check the workflow YAML and suggest the correction
- For `infra_failure`: no code fix — just explain what happened and suggest retrying

### 3. Pipeline Health Dashboard
- Across all repos in the org:
  - Success rate by pipeline (last 7/30/90 days)
  - Mean time to recovery (MTTR) by failure type
  - Top 10 flakiest tests (test name, failure count, last failure date)
  - Slowest pipelines (p50, p95 duration)
- Data sourced from GitHub API (workflow runs) and stored in BigQuery

### 4. Weekly Digest
- Automated weekly report posted to Slack or emailed to tech leads
- Contents: failure trends, new flaky tests, pipelines that degraded, and pipelines that improved
- Generated by a scheduled headless Gemini CLI run (cron job)

## How It Uses Gemini CLI

### Headless Failure Diagnosis

```yaml
# .github/workflows/diagnose-failure.yml
name: Diagnose Build Failure

on:
  workflow_run:
    workflows: ["CI"]
    types: [completed]

jobs:
  diagnose:
    if: ${{ github.event.workflow_run.conclusion == 'failure' }}
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - uses: google-github-actions/auth@v2
        with:
          workload_identity_provider: ${{ vars.GCP_WIF_PROVIDER }}
          service_account: ${{ vars.SERVICE_ACCOUNT_EMAIL }}
      
      - name: Download build log
        uses: actions/download-artifact@v4
        with:
          name: build-log
          run-id: ${{ github.event.workflow_run.id }}
      
      - uses: google-github-actions/run-gemini-cli@v1
        with:
          prompt: |
            Read the build log in build-log.txt.
            
            1. Find the root cause error (not warnings, not info messages)
            2. Classify it: code_error, test_failure, flaky_test, infra_failure, or config_error
            3. If code_error or test_failure: read the relevant source file and suggest a fix
            4. Post your analysis as a comment on the PR that triggered this build
            
            Format your response as:
            ## Build Failure Diagnosis
            **Classification:** [type]
            **Error:** [one-line summary]
            **Details:** [explanation]
            **Suggested Fix:** [code or config change]
```

### Custom Subagent: Log Analyzer

```yaml
# .gemini/agents/log-analyzer.md
---
name: log-analyzer
description: Analyze CI/CD build logs to extract the root cause error
model: gemini-3-flash-preview
tools:
  - read_file
  - grep_search
---
You are a build system expert. You read CI/CD logs.

## Rules
- Ignore INFO and DEBUG lines — focus on ERROR and FATAL
- Ignore npm install output unless it contains "ERR!"
- If you see "ECONNREFUSED" or "ETIMEDOUT", classify as infra_failure
- If you see "expected X but received Y", classify as test_failure
- If you see "error TS" or "SyntaxError", classify as code_error
- Always include the exact line number from the log where the error appears
```

## Data Model

```
pipeline_runs
├── id: UUID
├── repo: VARCHAR(255)
├── workflow_name: VARCHAR(255)
├── run_id: BIGINT — GitHub Actions run ID
├── status: ENUM(success, failure)
├── failure_type: ENUM(code_error, test_failure, flaky_test, infra_failure, config_error) NULLABLE
├── error_summary: TEXT NULLABLE
├── duration_seconds: INTEGER
├── triggered_by: VARCHAR(255) — commit SHA or PR number
├── diagnosed_at: TIMESTAMP NULLABLE
├── created_at: TIMESTAMP

flaky_tests
├── id: UUID
├── repo: VARCHAR(255)
├── test_name: VARCHAR(500)
├── test_file: VARCHAR(500)
├── failure_count_30d: INTEGER
├── last_failure: TIMESTAMP
├── quarantined: BOOLEAN DEFAULT false
└── updated_at: TIMESTAMP
```

## Acceptance Criteria

- [ ] A build failure triggers the diagnosis workflow within 60 seconds of completion
- [ ] The diagnosis correctly identifies the root cause error (not a downstream effect)
- [ ] Classification accuracy: >90% on a test set of 20 build logs with known failure types
- [ ] Flaky tests are identified by comparing the current run against the last 5 runs of the same test
- [ ] The weekly digest includes at least: success rate trend, new flaky tests, and top 3 slowest pipelines
- [ ] The log-analyzer subagent processes a 5,000-line log and returns results in under 10 seconds

## Tech Stack

- Pipeline: GitHub Actions
- Agent: Gemini CLI (headless mode)
- Analytics: BigQuery
- Notifications: Slack webhook or GitHub issue comments
- Dashboard: React + Recharts (or Looker Studio connected to BigQuery)
